プロジェクト概要

  • 本アプリは、英単語の「再確認コスト」を下げることを目的としたローカル動作の学習支援アプリです。
    単語帳アプリにLLMを組み合わせることで、単なる意味確認ではなく「例文生成」「翻訳評価」までを一つのアプリ内で完結させることを目指しました。
  • 個人の学習用途を想定しているため、コストをかけずに検証・改善を繰り返せることを重視し、無料枠で利用可能な範囲で設計しています。

想定ユースケース

  • 英語学習中に「この単語、前に見たな」と感じたときに、即座に確認・定着できることを想定しています。

システム構成図

  • 和訳・例文機能：テキスト入力→LLM→アウトプット→データベース登録
  • チャット機能：英単語入力→LLM→問題出題→評価→ヒントor合格出力

技術構成

  • Python
    • LLM API との親和性が高く、学習用途としても情報量が多いため選択しました。
  • gemini api
    • OpenAI APIやAzure OpenAI Serviceを検討しましたが、学習目的かつ個人開発であるため、コストをかけずに検証できる点を優先しました。
      さらに、チャット形式のやり取りと言語生成の両方を兼ね備えている点で選択しました。
  • tkinter
    • 環境構築が不要なデスクトップアプリで、exe化すれば配布もでき、Webアプリ化も検討しましたが、ローカルで完結する学習ツールを目指したため、このライブラリを選択しました。
  • sqlite3
    • 軽量かつ高速のデータベースであり、ローカルで動作するアプリのため選択しました。

課題認識

  • AI・自動運転の論文を読む過程で、英単語を調べ直す回数が多く、学習の流れが頻繁に途切れることに課題を感じました。
    既存の英単語帳やアプリでは「意味を調べる」「理解力の確認」の工程が分断されていたため、これらを一つの操作で完結できるアプリを自作しようと思いました。

設計判断

  • 翻訳APIなどある中でLLMを選択した理由
    • プロンプトで柔軟な対応が可能であるため
  • open AI APIやAzure OpenAI Serviceを選ばなかった理由
    • 無料でLLMを利用できるという点と対話形式のやり取りができることを優先しました。
  • 英文の評価を閾値で決めている理由
    • gemini apiではfunction_callingの機能を利用して入力に対してAIの返答を制御する予定でしたが、対話形式の機能とは同時に利用できないとの制約があり、英文の評価時に9点以上で「合格」、未満で「アドバイス」を出力するようにプロンプト入力で閾値制御をしました。
  • MVC構成を採用した理由
    • 個人開発ではあるが、業務での開発を想定し、UI・LLMロジック・データベース接続の責務を分離したかったです。
      翻訳評価ロジックやプロンプト設計が今後変更・改善される可能性が高く、UIやデータ管理への影響を最小限に抑える構成が必要だと判断しました。

構成・機能

  • ユーザーができること
    • 和訳を知りたい英単語を入力する
    • データベースへ「英単語」「和訳」「例文」の登録、削除、編集ができる
    • 登録した英語を閲覧するテーブル表示できる
    • AIとの翻訳の評価・アドバイスをチャット形式で行うことで単語の意味確認で終わらせず、実際に使って評価される流れを作ることで学習効率の向上を狙っています。

工夫した点・難しかった点

  • 検索窓の作成を工夫しました。テーブルのウインドウが表示された際にデータベースからデータを取得するのでそれに合わせた検索機能を実装しないといけない欠点がありました。そこでウインドウが表示されている時には最新のデータをリストで保持し、そのリストをコピーしコピーしたものを検索文字から文字列の前方一致で条件を付けてfor文で表示するように工夫しました。DBへの再クエリを毎回行う設計も考えましたが、パフォーマンスの低下を懸念してほかの手法を思考しました。



開発中に得た学び

  • API設計で.envファイルを用いて不正アクセスと管理のしやすさを学びました
  • MVCの型を利用し、運用保守や機能追加をしやすいよう構築できるようになりました。
